# CrustCore v0.2+ Roadmap — Track A (light it up), Track B (expand) & Track C (compose & adopt)

The v0.1 trusted core is **done and merged**: every capability pack already ships a std-only, fully-tested, transport-agnostic *decision core*, nano is **412.0 KiB** (51.5% of the 800 kB gate), and ~663 workspace tests pass green under `cargo xtask verify`. **Track A ("light it up")** wires real live I/O — model providers, Telegram, GitHub, MCP, subagent execution, native secret stores — onto those already-verified cores *without re-litigating the routing, policy, budget, or verifier logic they encode*. **Track B ("expand")** then adds net-new surfaces once the lit-up core is proven end-to-end. **Track C ("compose & adopt")** then adopts the ergonomics lessons of RIG and ADK-Rust — a unified provider registry, a `#[crust_tool]` macro, a typed workflow graph, a session/artifact service, RAG, OTel/GenAI telemetry, and a loopback dev UI — as non-nano sidecars/feature-gated packs that *consume the existing typed contracts unchanged*, never widening the trust boundary (v0.4, after Track A is lit up). This plan is authored to be executed by the **same parallel multi-worker supervisor/worker model the project itself implements** (CLAUDE.md §7): one supervisor integrates, a fleet of workers drafts patches in isolated worktrees, and only a `VerifiedPatch` ever ships.

## How to read this

This document has two layers. The **front matter** you are reading now — the dependency graph, the parallel execution model, the cross-cutting best practices, the v0.2 definition of done, and the milestone slices — is the contract every per-phase plan inherits; read it first and hold it in memory before opening any phase. The **per-phase detail sections** (P7-live, P8-store, P9-net, P10-net, P11-exec, P12-native, P13-net, P14-live, P5-join, the Track B phases, and the Track C phases (C1-providers, C2-toolmacro, C3-flow, C4-session, C5-rag, C6-telemetry, C7-devui)) are appended after this front matter and each follow the same shape: goal, leverage, prerequisites, a numbered task list with owned file globs, deferral boundary (CI-testable core vs live-gated part), contract-file impact, test/verify strategy, parallel-worker split, risks, definition-of-done checklist, and nano size impact. When a per-phase section and this front matter ever disagree, this front matter and the CLAUDE.md contract win.

## Dependency graph & execution waves

Every Track A phase has a **deterministic core already merged** in v0.1; its v0.2 work is the live-I/O layer plus any net-new surface. The waves below are derived purely from the *unmet* prerequisites of that live layer — a phase joins the earliest wave in which all of its prerequisites have landed. Phases inside a wave own disjoint file globs and run concurrently.

**Cross-cutting credential dependency (read first).** The secret broker (`crustcore-secrets`: `SecretBroker`, `ApprovedSecretView`, `CredentialProxy`, `Redactor`, `Tainted<T>`) **already exists and is a contract crate** — every live phase *consumes* it unchanged to get credentials to tools (never to the model, never to the sandbox env; invariants 1–3). What does *not* yet exist is at-rest native storage: **P8-store** adds OS keychains / encrypted-file vaults behind the existing `SecretStore` trait. Crucially, the broker's `InMemoryStore` plus an operator-supplied vault file is **enough to feed live credentials in dev and in the live-gated integration paths today** — so P8-store is a *hardening* dependency for production at-rest secrets, **not a hard blocker** for the wire adapters in P7-live / P9-net / P10-net / P13-net. Those phases can develop against `InMemoryStore` and adopt native backends as they land.

## Parallel multi-worker execution model

This plan is executed by the **exact supervisor/worker model CrustCore itself implements** (CLAUDE.md §7; `crates/crustcore-daemon/src/supervisor.rs` — the `AgentRegistry`, `Role`, budget-enforcing `Scheduler`, `Blackboard`, and `decide_integration`). Use it literally.

**Exactly one supervisor per build session.** Only the supervisor may: talk to the user, request approval, integrate patches / merge branches, push branches, open PRs, resolve secret handles for tools, spawn workers, and commit durable project state (`CHANGELOG.md`, contract docs). The supervisor is the only actor that holds authority — mirroring the runtime rule that only CrustCore authorizes, verifies, persists, exposes, or integrates.

**A worker fleet explores, drafts, and produces patches — never authority.** Each worker runs in its **own isolated git worktree** on a per-task branch `claude/<phase>-<slug>`, edits only the file globs assigned to it, runs its verifier in a sandbox, and reports a **structured result** (what changed, files touched, tests run, risks, open questions, and its proposed changelog lines) back to the supervisor via the blackboard. Workers never talk to the user (invariant 5), never edit each other's files, and never merge.

**Partition by disjoint owned file globs.** Two workers must never own overlapping paths; the supervisor assigns disjoint directories/crates up front. Each phase's detail section lists the globs it owns precisely so the supervisor can fan out without collision.

**Contract-file changes are serialized.** The contract files (CLAUDE.md, AGENTS.md, INVARIANTS.md, THREAT_MODEL.md, SECURITY.md, `docs/policy.md`, `docs/secrets.md`, `docs/sandbox.md`, `docs/backend-contract.md`, `crustcore-kernel/src/event.rs`, `crustcore-kernel/src/action.rs`, `crustcore-policy/src/decision.rs`, `crustcore-secrets/src/lib.rs`, `Cargo.toml`, `Cargo.lock`) are **never edited in parallel**. A phase that needs a contract change stops and routes it through a dedicated, maintainer-approved, single-PR serialized task *before* the dependent work — it does not bundle the change into unrelated work.

**Per-phase loop (every worker, every phase):**

```text
read the contract doc(s) for the phase
  -> implement the CI-testable deterministic core in the right crate
  -> launch a multi-agent adversarial review (each finding independently refuted)
  -> fix confirmed findings with regression tests
  -> cargo xtask verify GREEN (fmt, clippy -D warnings, tests, forbidden-deps, nano size gate)
  -> branch claude/<phase>-<slug>  (one task = one branch = one PR; declare owned globs)
  -> PR -> CI -> supervisor integrates -> merge
```

**Bounded budgets and concurrency.** Every spawned worker has wall-time, output-size, token/cost, and a concurrency cap (the `Scheduler`'s `max_concurrent`). Runaway fan-out is a modeled threat (budget exhaustion); keep concurrency bounded and purposeful. **One changelog writer at integration time:** workers report their changelog lines; the supervisor writes the consolidated `[Unreleased]` entries to avoid `CHANGELOG.md` merge conflicts.

**Worked example — fanning out Wave 1.** The supervisor opens Wave 1 by first landing the one serialized contract sub-PR it requires (the `Cargo.toml` workspace-dependency pins for P7-live's `tokio`/HTTP/TLS stack and P8-store's keychain crates — reviewed and merged alone). Then it spawns four workers on disjoint globs:

- **Worker A (P7-live):** owns `crates/crustcore-net/**` and `docs/model-routing.md`; implements the live `Provider` impls + Tokio/HTTP/TLS transport behind the existing `Engine`, gated behind a `live` feature; deterministic replay tests in CI, live calls behind `#[ignore]`.
- **Worker B (P8-store):** owns `crates/crustcore-secrets/src/store/**` (new backend module) and `docs/secrets.md` (serialized — routed as its own sub-PR); implements OS-keychain + encrypted-vault `SecretStore` impls behind the existing trait.
- **Worker C (P10-net):** owns `crates/crustcore-net/src/github/**` and `docs/github.md`; implements GitHub REST auth, branch push via the credential proxy, and draft-PR creation; mock-server contract tests in CI, live calls `#[ignore]`.
- **Worker D (P5-join):** owns `crates/crustcore-worktree/**` and `crates/crustcore-backend/src/verify*`; refines the integration-worktree join / verify-loop helpers; fully CI-testable on the local filesystem.

Workers A and C both touch `crustcore-net` but **own disjoint subtrees** (`crustcore-net/src/{lib,provider}` vs `crustcore-net/src/github`), so they do not collide; the supervisor confirms the partition before fan-out. Each worker returns a structured result; the supervisor reruns the full verifier on the integrated tree, writes the consolidated changelog, and merges in dependency order.

## Cross-cutting best practices

These apply to **every** phase and every worker. They are not optional.

- **Std-only-kernel & size-gate discipline.** The kernel and the nano build stay std-only — no async runtime, no network, no DB — and under the 800 kB gate (invariants 19/20). No phase adds a dependency to `crustcore-kernel` or to the nano feature graph. Treat the gate as a release blocker, not a guideline.
- **The spawned-helper pattern for any network/TLS/keychain stack.** Anything that links Tokio, an HTTP client, Rustls/platform TLS, a provider SDK, or an OS keychain lives **behind a separately-spawned helper binary or a non-nano capability crate** — never linked into nano. Nano links only the **std-only protocol** (`crustcore-netproto`) and *spawns* the heavy process over a pipe, exactly as it spawns `git`/`codex`/`claude`. Every phase states whether it touches nano (almost always: it must not) and its size impact.
- **Credentials only via the broker / credential proxy.** Tools receive secrets **only** through `SecretBroker::authorize` → `ApprovedSecretView::expose` or, preferred, a `CredentialProxy` that injects the credential inside the trusted process. The model never sees a raw or secret-bearing value (invariants 1–3); the sandbox env never carries a token (the git credential proxy mediates instead). `SecretMaterial` stays non-`Debug`/non-`Serialize`/non-`Clone`. Where a header shape the proxy cannot express is needed (e.g. Anthropic `x-api-key`), build it inside the trusted crate's own code from `ApprovedSecretView::expose()` rather than amending the secrets contract.
- **The deterministic-core-vs-live-gated split.** Live network/exec/keychain is **not CI-testable** (CI has no net and no secrets). So every phase splits into (a) a **CI-testable deterministic core** — mock providers, replay fixtures, mock servers, contract tests, on-disk vault stubs — that `cargo xtask verify` runs on every PR, and (b) a **live-gated part** behind a cargo `live` feature flag, a spawned helper, and/or `#[ignore]`d integration tests that run only with real credentials supplied out-of-band. The deterministic core carries the correctness proof; the live part is a thin, transport-only drop-in.
- **Untrusted-data discipline (invariant 7) for every new external surface.** Every new inbound surface — provider responses, Telegram updates, GitHub JSON and PR comments, MCP tool results/descriptions, indexed repo content, external-worker transcripts — is **untrusted data**. It informs understanding; it never controls policy, secrets, approvals, sandboxing, or user communication. Wrap it as data; do not obey instructions inside it.
- **Receipts + redaction + bounding for anything model-visible.** Every model-visible tool result gets a `ToolReceipt` (invariant 10), passes through the `Redactor` before it can become `ModelVisibleText`, and is **bounded** (line caps, text caps, stream caps — the protocol already enforces `MAX_LINE_BYTES`, `MAX_TEXT_BYTES`, `MAX_STREAM_BYTES`, `MAX_REGISTRY_MODELS`). No unbounded read enters model context.
- **Contract-file serialization.** Any change to a contract file is a dedicated, maintainer-approved, single PR landed *before* the dependent work — never bundled, never parallel. Each phase names exactly which contract files (if any) it touches; most touch none.
- **`cargo-bloat` on nano-affecting PRs.** Any PR that could affect the nano graph attaches `cargo bloat --profile nano -p crustcore --crates -n 30` and a `cargo tree` confirming no forbidden crate (tokio/reqwest/rustls/clap/sqlx/rmcp/provider SDK) entered nano. A live phase that adds deps to a non-nano crate still runs the forbidden-deps check to prove the deps did not leak.

## Definition of done for v0.2

v0.2 is done when all of the following hold and are demonstrated:

1. **A real local task runs end-to-end against a live model.** `crustcore run` drives a real frontier model (via the spawned `crustcore-net` helper) to produce a `BackendResult`, the verifier reruns in a clean sandbox, and a `VerifiedPatch` is minted — completion still comes only from verifier evidence (invariant 13).
2. **That `VerifiedPatch` opens a draft PR.** P10-net pushes the branch through the credential proxy and opens a **draft** PR from a `VerifiedPatch` only; merge still requires explicit approval; force-push stays denied by default.
3. **The full issue→PR golden task passes** against a mock GitHub server in CI and is demonstrated once live.
4. **Secrets come from a native store.** P8-store provides OS keychain / encrypted vault behind `SecretStore`; the broker resolves live credentials from it; no secret is serializable/debuggable/clonable or model-visible, and the redaction red-team fixtures pass.
5. **Telegram is operable as the default human channel** (invariant 15): allowlisted chat control, queue/steer, nonce-bound approvals, status — driven live through the net helper, with the model unable to send arbitrary Telegram text directly.
6. **Subagent execution works end-to-end** (P11-exec): the supervisor spawns real subagents in isolated worktrees, integrates only a verified merge, and every privilege is bound to the registry role, not a self-asserted claim.
7. **MCP tools call live through the gateway** (P13-net) with policy checks, receipts, redaction, and untrusted-data handling; unused servers cost zero model context.
8. **Nano is still under budget.** `crustcore-nano` remains < 800 kB stripped; the CI size gate is green; no live phase leaked an HTTP/TLS/keychain/provider stack into the nano graph (`cargo tree`/`cargo bloat` attached to each nano-touching PR).
9. **Every new external surface has a red-team fixture.** New prompt-injection, credential-exfil, fake-tool-result, MCP-hidden-instruction, and PR-comment-injection fixtures are added (or un-ignored) and pass; the existing red-team suite stays green.
10. **`cargo xtask verify` is green** across the workspace and the changelog records each phase's change, phase id, PR, owning role, size impact, and invariants touched.

## Suggested milestone slices

- **v0.2.0 — "first live patch & PR":** **P7-live + P10-net + P8-store**. The minimal slice that satisfies the headline DoD criteria 1–4: a live model produces a verified patch, it opens a draft PR through the credential proxy, and secrets come from a native store. This is the proof that the v0.1 trust story holds under real I/O end-to-end.
- **v0.2.x — "operable runtime & parallel orchestration":** **P9-net** (Telegram operable), **P13-net** (MCP live), **P11-exec** (subagent execution + integration verify), **P12-native** (native advisor/executor), **P14-live** (tree-sitter / persistent memory), **P5-join** (worktree/verify-loop join refinements). Lights up the remaining runtime surfaces and multi-agent orchestration on top of the v0.2.0 base.
- **v0.3 — Track B (expand):** the net-new surfaces — additional providers and meta-provider paths, the optional webhook server, further sandbox backends (Firecracker / container hardening), richer code intelligence, and broader multi-repo orchestration — built only after the lit-up Track A core is proven end-to-end and under budget.
- **v0.4 — "compose & adopt": Track C (C1–C7).** The ergonomic surfaces — unified multi-modal provider registry, `#[crust_tool]` macro, typed workflow graph, session/artifact service, RAG/vector-store pack, OTel/GenAI telemetry, and the loopback dev UI — adopted from RIG/ADK-Rust *after* the lit-up Track A core is proven end-to-end and under budget, each consuming the existing contracts unchanged, defaulting fail-safe, and shipping as a non-nano sidecar/feature-gated pack with zero nano impact. (Track C has its own dependency waves, cross-cutting principles, and definition of done in the **Track C — detailed surface plans** section below.)

## Track A — detailed phase plans (light it up)

### P7-live — Live model providers (OpenAI / Anthropic / OpenRouter / local OpenAI-compatible) in the spawned crustcore-net helper  _(Track A)_

**Goal.** Replace the deterministic MockProvider stand-in with real credentialed HTTP `Provider` adapters for OpenAI, Anthropic, OpenRouter, and local OpenAI-compatible endpoints (Ollama/vLLM/LM Studio), living entirely inside the *spawned* `crustcore-net` helper binary. The already-tested routing engine (dynamic registry/probe, Router/Budget/Reliable meta-providers, streaming sink, budget ledger) stays byte-for-byte unchanged: a live `Provider` drops into the existing `Vec<Box<dyn Provider>>` without touching routing logic. Credentials reach the outbound request only via the Phase-8 secret broker's credential-proxy pattern (`CredentialProxy::bearer` → `HeaderInjection`), never the model, never the sandbox env, never nano. Nano continues to link only `crustcore-netproto` (std-only) and spawn the helper over a pipe, so the 800 kB size gate and forbidden-deps gate stay green with zero nano size delta.

**Leverage.** Unblocks every model-consuming capability above it: the advisor/executor loop (docs/advisor-executor.md), subagent supervisor (P11), self-improvement evals (P15), and any role-driven routing become real instead of echoing canned text. It is the single seam that turns CrustCore from a verifier harness driven by mocks into one that can actually call a frontier model to produce patches — while preserving the entire trust story: completion is still verifier-owned (a model's output is just a BackendResult candidate; only a VerifiedPatch ships, invariant 13), and the no-secret-to-model guarantee (invariants 1-3) is enforced structurally because the key never leaves the trusted helper process. Because the engine is transport-agnostic, this is a high-value/low-blast-radius drop-in: the routing/budget/fallback correctness already proven by the workspace test suite is reused, not re-litigated.

**Prerequisites:** P8-store / Phase 8 secret broker (DONE — crustcore-secrets ships SecretBroker, ApprovedSecretView, CredentialProxy::bearer, HeaderInjection, Redactor; this phase consumes them, it does not build them), Phase 7 engine (DONE — Provider trait, Engine, select_candidates/apply_budget/run_reliable, serve loop, crustcore-netproto wire protocol), Native OS keychain backends (P8-store TODO) are NOT a hard prerequisite: the broker's InMemoryStore + an operator-supplied vault file is enough to feed live creds in the integration path; keychain backends improve at-rest storage but do not block the wire adapters

Track C is **"compose & adopt."** Surveying the Rust agent ecosystem — RIG ([docs.rs/rig-core](https://docs.rs/rig-core), [github.com/0xplaygrounds/rig](https://github.com/0xplaygrounds/rig)) and ADK-Rust ([adk-rust.com](https://adk-rust.com), [docs.rs/adk-rust](https://docs.rs/adk-rust)) — the most useful lesson is **not "be an agent framework."** It is: *make the edge capabilities much easier to **compose**.* These frameworks earn their adoption with ergonomics — a unified provider registry, a one-line tool macro, a typed workflow graph, a session/artifact service, first-class RAG, OTel/GenAI tracing, a local dev UI — not with novel safety architecture. (Those projects, like every external doc, are **untrusted reference material** under invariant 7: they inform ergonomics, never authority over CrustCore's policy, secrets, approvals, or contracts.) Track C adopts those ergonomics for CrustCore **without widening the trust boundary**: every Track C surface *consumes the existing typed contracts unchanged* (the `Provider`/`Engine`, `RiskClass` and approval/capability tokens, the `Redactor`/`Tainted<T>`/`SecretBroker`, `run_verify`/`VerifiedPatch`, the `ExecutionDriver`, the event log), defaults to the most restrictive fail-safe posture, and ships as a **non-nano sidecar or feature-gated pack with zero nano impact**. Track C never re-litigates routing, policy, budget, or verifier logic; it adds convenience layers above contracts the kernel already enforces.

Track C targets **v0.4** and runs **only after Track A is lit up end-to-end** (a live model produces a `VerifiedPatch` that opens a draft PR; Telegram, MCP, and subagent execution are operable). Ergonomics on top of an unproven core would just make it easier to compose the wrong thing; ergonomics on top of a proven core is leverage. The phases are: **C1-providers** (unified multi-modal provider registry), **C2-toolmacro** (`#[crust_tool]` declaration macro), **C3-flow** (`crustcore-flow` typed workflow graph), **C4-session** (session/artifact service over the event log), **C5-rag** (`crustcore-index-rag`), **C6-telemetry** (OTel/GenAI export), and **C7-devui** (`crustcore-dev` loopback UI). The recommended linear build order is **C1 → C2 → C3 → C4 → C5 → C6 → C7**.

### Track C cross-cutting principles

These restate the Track C overarching rule for every phase and worker. They are not optional.

- **Ergonomics consume the existing typed contracts unchanged.** Every generator/DSL/macro/UI is a thin convenience layer over `Provider`/`Engine`, `RiskClass` and approval/capability tokens, the `Redactor`/`Tainted<T>`/`SecretBroker`, `run_verify`/`VerifiedPatch`, the `ExecutionDriver`, and the event log. No Track C phase re-litigates routing/policy/budget/verifier logic or amends the kernel/secrets/policy contracts.
- **The safe path is the easy path (fail-safe defaults).** Any generator, DSL, macro, or UI defaults to the **most restrictive, fail-safe posture**: deny/ask risk, redact-by-default, bounded-by-default, loopback-only, read-only-by-default. A forgotten classification fails **closed**, never open — a tool with no declared `RiskClass` is treated as the highest risk; a UI with no auth binds to loopback only. **Critically, no ergonomic layer can be a side-effect or completion path:** a `#[crust_tool]` macro expansion, a `crustcore-flow` node, a session entry, or a dev-UI action **routes every side effect through `crustcore-policy` (invariant 8) gated on approval/capability tokens (invariants 4, 14) and through an explicit sandbox profile (invariant 9), and may reach completion only via `crustcore-backend::verify::run_verify` — the sole `VerifiedPatch` minter (invariant 13)**. No Track C surface may construct an `Approved<T>`, capability token, or `VerifiedPatch`; any "macro/node/UI/tool output → approval/side effect/completion" path is a bug and fails the red-team. Untrusted upstream output (tool/model/MCP/repo) stays `Tainted`/data (invariant 7) and is never promoted to a control decision.
- **Every surface is a non-nano sidecar / feature-gated pack with zero nano impact.** No Track C code enters `crustcore-kernel` or the nano feature graph. Each new crate states its **tier** (sidecar/pack, **1–10 MB**, *never* the flagship size claim) and proves a **zero nano delta** via `cargo tree` (no forbidden crate — tokio/HTTP/TLS/SDK/keychain/web-server/proc-macro runtime/vector-store — entered nano) and `cargo bloat` (no size change), attached to the PR. Proc-macro crates (C2) are build-time only and link no runtime into nano.
- **Every new surface gets a red-team fixture proving no bypass.** Each ergonomic layer ships a deterministic CI red-team fixture demonstrating it **cannot** be used to bypass policy, approval, sandbox, the verifier, or the no-secret-to-model guarantee — e.g. a generated tool cannot escape its `RiskClass`, a workflow node cannot mint a `VerifiedPatch` without `run_verify`, the dev UI cannot expose a secret, approve a side effect, or act as a user/subagent comms channel (invariants 5, 16).

### Definition of done for Track C (v0.4)

Track C is done when all of the following hold and are demonstrated:

1. **Each of C1–C7 ships its CI-testable deterministic core** — mocks, fakes, replay fixtures, on-disk stubs — run by `cargo xtask verify` on every PR with no network or secrets.
2. **Each of C1–C7 ships a live/feature-gated part** behind a cargo feature + `#[ignore]`d integration tests, exercised only with real creds/binaries out-of-band.
3. **Each of C1–C7 ships a red-team fixture** proving the ergonomic layer cannot bypass policy, approval, sandbox, the verifier, or the no-secret-to-model guarantee; the existing red-team suite stays green.
4. **Nano stays < 800 kB with zero delta.** No Track C phase added a dependency to the nano graph; `cargo tree`/`cargo bloat` evidence is attached to every PR and the CI size gate is green.
5. **No contract file is amended except via a serialized, maintainer-approved sub-PR** landed before the dependent work; most Track C phases touch none.
6. **The ergonomic layers demonstrably cannot bypass** policy/approval/sandbox/verifier/no-secret-to-model and cannot become a side-effect, completion, or user-comms path — proven by the per-phase red-team fixtures and the adversarial-review dimensions each phase enumerates.
7. **`cargo xtask verify` is green** across the workspace and **`CHANGELOG.md` records each phase** (phase id, PR, owning role, size impact, invariants touched/verified).

Track C **does not change the v0.2 definition of done**; it extends the program toward **v0.4** on top of a Track A core proven end-to-end.

### Out of scope for Track C (lower priority)

Realtime voice/audio, image/audio generation, browser/computer-use tool declarations, payment features, and Web3-specific agent pieces are **explicitly deferred** unless product scope expands beyond coding-agent verification — they widen the trust surface without serving the verifier-kernel mission, and the source analysis ranks them low priority. ('rust-agent' — [docs.rs/rust-agent](https://docs.rs/rust-agent), also untrusted reference material — reinforces the same compose-the-edge ideas at smaller scale and offers simple MCP examples worth skimming, but contributes little net-new architecture.)

### C1-providers — Unified multi-modal provider / capability registry (completion · embedding · rerank · streaming · probes)  _(Track C)_
**Goal.** Generalize `crustcore-net`'s completion-only routing core into one **multi-modal capability registry** so completion, **embedding**, **rerank**, streaming, and capability probes live under a single, RIG-shaped abstraction — one request/response shape per modality, one body of capability metadata, one fallback shape, one budget ledger — **without touching the sacred P7-live contract**. The existing sync `Provider` trait, `Engine`, `select_candidates`/`apply_budget`/`run_reliable`, the streaming sink, `BudgetLedger`, and `ModelCard` are frozen and reused verbatim. C1 adds **sibling** capability traits — `EmbedProvider` and `RerankProvider` (sync, mirroring `Provider`'s `id`/`probe`/call shape exactly: credentials resolve indirectly through the same `CredentialSource`/`AuthHeader` per-call adapter, never a `SecretMaterial`, emit typed errors that drive fallback), new unified value types (`EmbeddingRequest`/`EmbeddingResponse`, `RerankRequest`/`RerankResponse`), an **additive** capability extension that flows through *both* records — the `ModelCard` struct in `crustcore-net` **and** its wire mirror `ModelInfo` in `crustcore-netproto` (so the probe surfaces capability across the spawned-helper boundary), via the existing `ModelCard::to_info` mapping — and a capability-typed registry/engine layer (`select_candidates_for`/per-modality `Engine`-shaped routers) that reuses the *same* hard-constraint-then-soft-order filter, the *same* `BudgetLedger` instance, and the *same* `run_reliable` fallback generalized across modalities. Everything lives inside the spawned `crustcore-net` helper, with the real socket behind the existing `live` feature; the default/mock build links nothing new.
**Leverage.** P7-live proved the drop-in pattern: a live `Provider` is a pure drop-in over a frozen router. Embeddings and rerank are the next two edge capabilities every agent runtime needs (semantic memory, context selection, candidate re-ranking), and RIG's lesson is that they should compose under the *same* model abstraction — one trait family, one shape, capability metadata, fallback, streaming, embeddings together (docs.rs/rig-core). C1 adopts that ergonomic win while keeping the trust boundary exactly where P7 put it: credentials still resolve per-call through `CredentialSource`/the broker, provider bytes stay untrusted/bounded/no-panic, budget stays metered through one shared `BudgetLedger`, and capability is probed with conservative-off defaults — never re-litigated. Concretely, this phase is the **producer** that `B3-vector-memory` consumes: B3's "embedding provider over the net helper" is exactly C1's net-side `crustcore-net::EmbedProvider` routed by this registry, which becomes the live backend that fulfils B3's *consumer-side* `crustcore-index::embed::Embedder` trait (the two are deliberately distinct: C1 owns the routed *provider* trait in `crustcore-net`; B3 owns the *consumer* trait in `crustcore-index`, which C1 supplies vectors to rather than redefines — stated explicitly so B3 builds **on** it, not beside it; C1 extends the routing core and does not duplicate B3's store/semantic-select). It also unblocks rerank-assisted context selection in `crustcore-index::select_context` without that crate ever linking HTTP/TLS.
**Prerequisites:** P7-live (DONE — `Provider`/`Engine`/`select_candidates`/`apply_budget`/`run_reliable`/streaming sink/`BudgetLedger`/`ModelCard`/`transport::HttpClient`+`ReplayClient`+`UreqClient`/`credsource::CredentialSource`+`AuthHeader`+`AuthStyle`/`providers::map_status_error`; **hard blocker** — C1 generalizes its registry), `crustcore-netproto` (the std-only wire protocol nano spawns, with the `Request`/`Response` enums, `ModelInfo`, `serve`, and the `MAX_*` bound family; **hard blocker** — needs additive request/response variants + additive `ModelInfo` capability fields + new `MAX_BATCH`/`MAX_DOCS` consts), the P8 secret broker (`crustcore-secrets`, soft — only the live `CredentialSource` binding), and `crustcore-secrets::Redactor` (soft — for any model-visible rerank scores/diagnostics). `B3-vector-memory` and `crustcore-index` rerank adoption are **downstream consumers**, not blockers.

- `select_candidates_for` + per-modality engines route over the **same** hard-constraint filter, the **same** single `BudgetLedger` instance, and a `run_reliable`-shaped fallback; CI tests cover routing, budget accumulation across modalities, capability-missing fail-closed, and safe fallback.
- Live OpenAI-compatible `/v1/embeddings` and a `/v1/rerank` adapter are implemented over `transport::HttpClient`, fully tested in CI with `ReplayClient`/mocks; the real `UreqClient` path is `live`-gated only.

### C2-toolmacro — `#[crust_tool]` tool-authoring macro (schema · bounded I/O · redaction · fail-safe risk defaults · fixtures)  _(Track C)_
**Goal.** Make authoring a *safe* capability-pack tool trivial without widening the trust boundary. Introduce a compile-time-only proc-macro crate `crustcore-tool-macro` and a tiny std-only runtime support crate `crustcore-toolkit` (the traits the generated code targets). Annotating a function or struct with `#[crust_tool]` GENERATES, mechanically: (1) a JSON-Schema description of the tool's typed params and result (derived from the Rust signature, not hand-written), surfaced through a `CrustTool::schema()` method on the `crustcore-toolkit` trait; (2) bounded input/output handling that reuses the existing caps (`crustcore_types::BoundedText::DEFAULT_MAX` / `BoundedText::truncated` and the secrets `MAX_*` constants) and refuses-on-overrun (an explicit `ToolError::InputTooLarge` / `OutputTooLarge`) — never an unbounded read into model context; (3) a redaction step so every model-visible result is run through `crustcore_secrets::Redactor::to_model_visible` (the *sole* constructor of `ModelVisibleText`) and only emerges as `ModelVisibleText` (invariants 2, 10), never as a raw `String`; (4) `crustcore_receipts::ToolReceipt` wiring so each model-visible result is receipted (invariant 10): the macro emits a *call site into the trusted host* that owns the `crustcore_receipts::MacKey` and calls `ReceiptChain::mint(&ReceiptParams{ args, result, .. })` over the **exact redacted-and-bounded bytes shown to the model** — generated code never holds a `MacKey`, never calls `mint` itself, and the receipt's `result_hash` therefore binds the post-redaction bytes; (5) a **fail-safe risk default**: the generated `default_reversibility()` returns the most restrictive `crustcore_types::Reversibility::Destructive`, which `PolicySnapshot::classify` maps to `RequireApproval`/`Deny` (and to `Deny` outright under `RiskProfile::ReadOnly`), so a tool author must *explicitly* downgrade via `#[crust_tool(reversibility = "Reversible")]` and a *forgotten* classification fails closed; the macro never selects the host's `RiskProfile` — that is host config; (6) generated `#[test]` fixtures asserting schema validity, bound enforcement, redaction-before-visibility (including a `Redactor::would_leak` defense-in-depth check), and the fail-safe default. The macro emits *wiring* that calls into `crustcore-policy`'s `PolicySnapshot::classify` and `crustcore-secrets`'s `Redactor` unchanged — it never embeds its own allow/deny logic, never inlines a decision, and never lets generated code construct an `Approved<T>` (whose sole constructor is `AuthorizedUser::approve`), so generated code cannot self-authorize (invariants 4, 8).
**Leverage.** Hand-written tools today (`crustcore-worktree::tools`, the `crustcore-mcp` server tool path, daemon helper tools) each re-implement the same five-step safety dance — schema, bound, redact, receipt, classify — by hand, which is exactly where a tired author forgets a redaction or defaults a risk to "reversible." `#[crust_tool]` collapses that dance into a derive while making the **safe path the easy path** (Track C principle P2): the generated default is the most restrictive, redaction and bounding are non-optional, and there is no generated code path that produces a model-visible `String` bypassing the `Redactor`. This unblocks fast, *uniformly safe* authoring of mcp/daemon/backend pack tools and future Track C surfaces, and it preserves the trust story because the generated code is a pure consumer of the already-verified policy/secrets/receipts contracts — it adds an ergonomic layer, it does not re-litigate any decision.
**Prerequisites:** `crustcore-policy` (`PolicySnapshot::classify`, `PolicyDecision`, `RiskProfile`, `caps::Approved`/`AuthorizedUser::approve`) and `crustcore-types::Reversibility` — DONE; `crustcore-secrets` (`Redactor::to_model_visible`, `Redactor::would_leak`, `ModelVisibleText`, `BoundedText`) — DONE and nano-linked; `crustcore-receipts` (`ToolReceipt`, `ReceiptChain::mint`, `ReceiptParams`, `MacKey`) — DONE; the hand-written `crustcore-worktree::tools` and `crustcore-mcp` tool paths as the migration reference. Soft: C1 (if a shared toolset/registry surface lands there, `crustcore-toolkit` should expose its trait in a compatible shape). **Reconciliation:** C2 *extends/standardizes* the per-tool boilerplate that P11-exec subagents and the MCP server path already write by hand — it does **not** change subagent execution (P11-exec) or the MCP gateway (`gateway_check`/`filter_result`); a `#[crust_tool]` tool's runtime trust still flows through those unchanged. No hard blocker beyond the merged cores.

### C3-flow — crustcore-flow — typed workflow graph (model · tool · verify · review · parallel · loop_until · route · join)  _(Track C)_
**Goal.** Add a new non-nano sidecar crate `crustcore-flow` that gives an ergonomic, **typed workflow DSL** over CrustCore's existing supervisor/subagent/verify primitives — ADK-Rust's workflow variety (sequential, parallel, loop, graph, router; see adk-rust.com) made composable, without widening the trust boundary. A `Flow` is a directed graph of typed nodes: `model` (consult a model role through the spawned `crustcore-net` helper), `tool` (invoke a `crustcore-policy`-gated tool), `verify` (rerun the user verify command via the public `crustcore_backend::verify::run_verify`), `review` (an advisory `crustcore-daemon::advisor` check), `parallel` (bounded fan-out), `loop_until` (bounded iteration with a predicate), `route` (router/branch over typed state), and `join` (merge child results). Graph evaluation is **deterministic given node results**: the engine (`FlowEngine`) is a pure scheduler that owns no I/O — every effectful node delegates to an **injected driver** (the project's own seam pattern, inspired by the closure-injection used inside `crustcore_backend::verify` and by `crustcore_daemon::exec::SubagentExecutor`/`run_subagent`). So the whole graph is unit-testable with fakes in CI, and live transports drop in behind a `live-flow` feature.
**Leverage.** Today composing a multi-step agent run (consult → implement → verify → review → integrate) means hand-wiring the supervisor, `run_subagent`, `run_verify`, and `decide_integration` per task. `crustcore-flow` turns that into a declarative, typed plan that *consumes the existing contracts unchanged* — it never re-litigates routing, policy, budget, or the verifier, and it is **a planning/scheduling layer, not an integration authority**: `crustcore_daemon::supervisor::decide_integration` remains the sole integration decision and the flow itself never opens a PR or merges. The trust story is preserved structurally: a `Flow` is a **plan, not an authority**. The only completion path is a `verify` node whose driver calls the public `run_verify` (invariant 13, the sole `VerifiedPatch` minter); `model`/`review`/`tool` output is advisory/result data that can never mint `Approved<T>`, reach the user, or steer a side effect (invariants 4, 5, 6, 7). This unblocks C-track ergonomics (a builder DSL, later a visual graph) and gives the daemon a single, auditable representation of "what the agent is about to attempt" while leaving `decide_integration` and the supervisor's authority exactly where they are.
**Prerequisites:** the public `crustcore_backend::verify::run_verify` (DONE, the sole `VerifiedPatch` minter — note `run_verify_with` is a private closure-injection helper inside `crustcore-backend`, NOT a cross-crate API; flow calls `run_verify`); `crustcore_daemon::exec::{SubagentExecutor, run_subagent}` + P11-exec (DONE — subagent execution + integration-worktree verify) as the live `model`/`tool` driver; `crustcore_daemon::supervisor::{Role, Blackboard, decide_integration}` (DONE) which this composes and leaves as the sole integration authority, never replaces; `crustcore_daemon::advisor::{Advisor, AdvisorNote, NativeAdvisor, should_consult}` (DONE) for `review`; `crustcore-policy` (`RiskClass`, `Approved<T>` gated by `AuthorizedUser`, `decide`) and `crustcore-sandbox` for `tool` nodes; `crustcore-secrets` (`Tainted<T>`, `Redactor`) for the predicate boundary; the `crustcore-net` sync `Provider`/`Engine` (P7-live DONE, trait left UNCHANGED) reached only through the spawned helper. Hard blockers: P11-exec and the verify core. Soft: C1/C2 Track C ergonomics phases (shared builder conventions) if they land first.

### C4-session — Conversation / session / artifact service (SessionId · ConversationId · snapshots · resumable runs · history compaction)  _(Track C)_
**Goal.** Add a NEW non-nano crate `crustcore-session` that gives the daemon, `crustcore-flow` (C3), and the dev UI (C7) the application-level session model CrustCore lacks today — adopting ADK's session/state/artifacts/event-stream shape and RIG's conversation-memory-with-history-shaping ergonomics — **as a VIEW over the hash-chained event log, never a competing store.** It provides typed `SessionId` / `ConversationId`, derived **state snapshots**, opaque **artifact handles**, **resumable runs**, and **history compaction** policies. The crucial design rule, restated everywhere it can be enforced structurally: the `crustcore-eventlog` append-only chain (`EventLog`, `EventFrame`, `FrameMeta`, `EventSeq`) is the single source of truth. A session is an *index* over frames; a snapshot is a *derived projection* at a known `seq`; a resumable run is a *replay-and-verify* of the relevant frame range. Nothing here mints kernel state, mutates the chain, or persists secret material. **Explicit non-goal (invariant 13):** a `SessionView`/`SessionService` NEVER completes a task, integrates a patch, opens a PR, or mints a `VerifiedPatch` — completion remains solely `crustcore_backend::verify::run_verify`; resume reconstructs a *view* and nothing more. The facade exposes no method returning `Approved<T>`, a capability/approval token, a `VerifiedPatch`, or any side-effect trigger.
**Leverage.** This is the connective tissue that lets every higher Track C surface (flow graphs, the dev UI, multi-turn conversations) address "a run" by id, resume it, and shape its history for the model — without re-inventing storage and without widening the trust boundary. Because a session is a read-only projection, the trust story is preserved end to end: resuming reuses the event-log verifier (`EventLog::verify` / `verify_to_head`) and the receipt↔log join (`crustcore_receipts::join::verify_against_log`), so a tampered or broken log **refuses to resume** rather than silently reconstructing a corrupted view (invariants 7, 18). Snapshots are derived only from frames explicitly marked `Visibility::ModelVisible` (any `Internal`/unclassified frame fails closed out of a model-visible projection), then each field is passed through the `Redactor`; because `SecretMaterial` deliberately implements none of `Debug`/`Clone`/`Serialize` and has no conversion to `ModelVisibleText` (invariant 3), a secret *cannot* be captured into a snapshot — the guarantee is a type fact, not a runtime check. Compaction reuses `crustcore-index`'s redact-then-bound posture, its `MAX_*` caps, and the `ModelVisibleText` never-authority tag so compacted history is small, bounded, and never authoritative (invariants 7, 11).
**Prerequisites:** `crustcore-eventlog` (DONE — `EventLog`, `EventFrame`, `FrameMeta`, `EventSeq`, `ChainStatus`/`ChainStatus::is_intact`, `verify`/`verify_to_head`, `iter`, `inspect`, `export_jsonl`, `RedactionState`); `crustcore-receipts::join::verify_against_log` + `FrameRef` + `JoinStatus`/`JoinStatus::is_joined` + `JoinBreak` (P5-join, DONE) for the resume-time receipt cross-check; `crustcore-secrets` `Redactor` + `ModelVisibleText` + non-`Serialize` `SecretMaterial`/`Tainted<T>` (DONE) for snapshot redaction; `crustcore-kernel` `Visibility` (`ModelVisible`/`Internal`) frame gating and `EventKind` variants (`UserMessageQueued`/`ModelOutputReceived`/`ToolCallCompleted`/`JobLeased`/`TaskKilled`/`TaskFailed`); `crustcore-types` `ArtifactId([u8;32])`, `TaskId`, `JobId`, `EventSeq` for opaque handles; `crustcore-index` `select_context`/`semantic_select` posture + `MAX_*` caps (`MAX_CONTEXT_BUNDLE`/`MAX_CONTEXT_FRAGMENTS`/`MAX_FRAGMENT_BYTES`, DONE/B3) reused by compaction; kernel lease/heartbeat/cancellation/recovery semantics (DONE) reused by resumable runs. Soft consumers, not blockers: `crustcore-flow` (C3) and the dev UI (C7) — this crate ships standalone with local fixtures. Hard blocker: none beyond the DONE cores above.

### C5-rag — crustcore-index-rag — RAG + vector-store adapter pack (chunker · symbol-aware metadata · query planning)  _(Track C)_

### C6-telemetry — OpenTelemetry / GenAI-semantic-convention export from the audit log (read-only projection)  _(Track C)_
**Goal.** Add a new non-nano sidecar crate `crustcore-telemetry` that projects CrustCore's already-authoritative audit trail — the hash-chained event log (`crustcore-eventlog`: the `crustcore_kernel::EventKind` set carried in binary frames, with per-frame `visibility: Visibility` and `redaction: RedactionState`) and the MAC-chained tool receipts (`crustcore_receipts::ToolReceipt`, cross-checked against the log by `crustcore_receipts::join::verify_against_log`) — into standard OpenTelemetry spans and metrics under the GenAI semantic conventions. RIG advertises GenAI-semconv compatibility and ADK ships telemetry; CrustCore already has a *stronger* audit log, so telemetry is not a new source of truth but a **derived, read-only projection** of frames CrustCore already minted. The crate maps frame kinds to spans (`EventKind::ModelRequestStarted`/`ModelOutputReceived` → `gen_ai.*` model-call spans; `EventKind::ToolCallStarted`/`ToolCallCompleted` + the matching receipt → `crustcore.tool` spans; `EventKind::PatchProposed`/`PatchVerified`/`PatchRejected` → `crustcore.verify` spans carrying the `verify::run_verify` outcome; `EventKind::JobLeased`/budget-bearing frames → `crustcore.budget` metrics) and exports them over OTLP. The OTel/OTLP SDK is heavy (Tokio/HTTP/protobuf), so the exporter is feature-gated and runs spawned/sidecar — **never linked into nano**, which keeps its own std-only event log as the one source of truth.
**Leverage.** This turns CrustCore's existing receipts-grade audit chain into something every observability stack already speaks — model calls, tool runs, verification outcomes, and budget burn become standard GenAI spans/metrics in Grafana/Honeycomb/Jaeger — *without* widening the trust boundary one inch. The projection mints nothing and changes no state (the same read-only posture as P5-join's receipt↔log cross-check, which it consumes rather than duplicates), so the audit log stays authoritative and telemetry is explicitly never authoritative: a deleted or altered span cannot affect a verdict, a budget, or a `VerifiedPatch` (which only `verify::run_verify` may mint). Because every emitted attribute value and metric label is scrubbed through `crustcore_secrets::Redactor::redact` before it leaves the process, the outbound surface inherits the no-secret-to-model / no-secret-to-log guarantee (invariants 1–3) rather than re-deriving it.
**Prerequisites:** `crustcore-eventlog` (DONE — `EventKind` frames with `visibility`/`redaction` fields, `EventLog::iter() -> FrameIter`), `crustcore-receipts` + `join::verify_against_log` (DONE — `ToolReceipt`, P5-join, which C6 consumes for the receipt↔span binding and does not re-implement), `crustcore-secrets::Redactor` + `Tainted<T>` (DONE — outbound redaction), `crustcore-backend::verify::run_verify` outcome shape (DONE — for verify-span attributes), and the `SecretBroker` for any OTLP endpoint auth (DONE). Hard blockers: eventlog + receipts + `Redactor`. Soft: C1 ergonomics facade (if it exists, telemetry can reuse its config surface), and B6 release infra (for the feature matrix). No Track A/B phase blocks this.
