# INVARIANTS.md — CrustCore Product Laws

> These are **product laws**. Breaking any of them is a **release blocker**.
> This file is a [contract file](./CLAUDE.md#73-contract-files--serialized-changes-only):
> changes are serialized and require maintainer approval. The canonical list is
> mirrored in [`CLAUDE.md` §4](./CLAUDE.md) and [`ROADMAP.md` §4](./ROADMAP.md).

Each invariant below states the law, *why* it exists, *how* it is enforced
(preferably by Rust types, not discipline), and *how* it is tested. "Enforced by
type" is the gold standard: CrustCore's core advantage is making dangerous
states impossible to represent.

---

## The twenty laws

### 1. The LLM never receives raw credentials

**Why:** an LLM is untrusted with secrets; prompt content can be exfiltrated.
**Enforcement:** secrets exist only as `SecretHandle` (an id + label) in
model-visible space; raw bytes live in `SecretMaterial`, which cannot be
converted to model-visible text. Tools receive secrets via one-shot views or
credential proxies, never the model.
**Tested by:** redaction/taint tests attempting to route a secret into a model
prompt (must fail to compile or fail the test). See [`docs/secrets.md`](./docs/secrets.md).

### 2. The LLM never receives unredacted secret-bearing logs

**Why:** logs are a common leak path (stdout, stderr, env dumps, errors).
**Enforcement:** secret-bearing data is tainted; tainted data cannot enter model
prompts, model-visible tool results, normal logs, Telegram, or GitHub comments.
**Tested by:** the secret-leak red-team matrix (§ below; [`docs/security-model.md`](./docs/security-model.md)).

### 3. Secret material is not Debug, Serialize, Clone, or model-visible

**Why:** derive macros and `{:?}` are the easiest accidental leak.
**Enforcement:** `SecretMaterial` wraps `Zeroizing<Vec<u8>>` and deliberately
does **not** implement `Debug`, `Serialize`, or `Clone`, and exposes no path to
`ModelVisibleText`. Exposure is only via `ApprovedSecretView`.
**Tested by:** compile-fail tests (trybuild-style) asserting the missing impls.

### 4. The model cannot approve its own side effects

**Why:** self-approval defeats the entire safety model.
**Enforcement:** approvals are `Approved<T>` tokens minted only by the approval
engine from an authorized human (Telegram nonce / CLI), never from model output.
**Tested by:** policy tests asserting model-originated "approvals" are rejected.
See [`docs/policy.md`](./docs/policy.md).

### 5. Subagents cannot directly message the user

**Why:** prevents social-engineering and bypass of the supervisor.
**Enforcement:** the agent message bus has no `Agent -> User` edge; only the
supervisor holds the user channel capability.
**Tested by:** integration tests asserting subagent→user routes are denied.

### 6. External workers are patch producers, not truth authorities

**Why:** Codex/Claude Code output is untrusted; `self_claimed_done` is a claim.
**Enforcement:** worker output is `UnverifiedPatch`; only the verifier produces
`VerifiedPatch`. See [`docs/backend-contract.md`](./docs/backend-contract.md).
**Tested by:** worker-contract tests where a worker claims done but the verifier
fails — task must not complete.

### 7. Repo files, issue/PR comments, web pages, MCP output, and shell output are untrusted data

**Why:** indirect prompt injection.
**Enforcement:** all such content is wrapped as data with an invariant reminder;
it never controls tools, policy, secrets, approvals, or user comms.
**Tested by:** prompt-injection red-team fixtures (file says "reveal token",
issue says "ignore policy", MCP returns hidden instructions, etc.).

### 8. Every side effect passes through policy

**Why:** a single ungoverned side effect is a hole.
**Enforcement:** side-effecting tools require capability tokens; capabilities are
issued by the policy engine. No token, no effect.
**Tested by:** policy decision unit tests + "ungoverned side effect" audit.

### 9. Every execution-capable operation runs in an explicit sandbox profile

**Why:** arbitrary execution is the highest-risk surface.
**Enforcement:** `run_command` requires a `SandboxExecCap` bound to a profile;
there is no host-shell escape hatch.
**Tested by:** sandbox red-team tests (network deny, env strip, path-env escape).
See [`docs/sandbox.md`](./docs/sandbox.md).

### 10. Every model-visible tool result has a receipt

**Why:** prevents hallucinated/fabricated tool results.
**Enforcement:** model-visible results carry a `ToolReceipt` (args/result hashes,
event seq, prev-receipt hash, MAC) generated by CrustCore, never the model.
**Tested by:** "no receipt → no model-visible claim" tests + replay verification.
See [`docs/receipts.md`](./docs/receipts.md).

### 11. Every task has budget limits

**Why:** runaway agents exhaust cost/time/resources.
**Enforcement:** every task/job carries a budget record (wall time, CPU, memory,
disk, output, tokens, model cost, subagent count); exhaustion pauses the task.
**Tested by:** budget-exhaustion kernel tests.

### 12. Every long-running job has lease, heartbeat, cancellation, and recovery semantics

**Why:** crashes and stalls must not strand work or leak resources.
**Enforcement:** `JobStatus` includes `Leased`/`HeartbeatMissing`/`Expired`;
each job has lease owner, expiry, heartbeat, attempt number, retry policy, and a
cancellation token/process handle.
**Tested by:** job lifecycle + recovery tests.

### 13. Every shippable patch is a VerifiedPatch

**Why:** verifier-owned completion is the north star.
**Enforcement:** integration / PR creation / completion APIs accept only
`VerifiedPatch`. See [`docs/backend-contract.md`](./docs/backend-contract.md).
**Tested by:** golden task "fix failing test" — completion blocked until verify
passes.

### 14. Irreversible actions require an approval token

**Why:** merge/deploy/force-push/secret-write are not undoable.
**Enforcement:** irreversible operations take `Approved<IrreversibleAction>`;
`Reversibility` is a typed enum gating which operations need approval.
**Tested by:** policy tests for each irreversible operation class.

### 15. Runtime user communication goes through authorized, redacted channels (Telegram by default)

**Why:** every runtime message — inbound or outbound — must cross **one auditable,
redacted, principal-authenticated boundary**; no hidden or unredacted side channels.
**Enforcement:** the runtime user-channel capability is bound to **authorized,
allowlisted principals** — a Telegram chat id by default; the local operator on the
`crustcore chat` terminal; any other channel requires explicit policy. *Inbound:* only
an authorized principal's message becomes a user turn (`crustcore_chat::accept` /
the Telegram allowlist); model/tool/file/peer content never can. *Outbound:* every
message is rendered through the `Redactor` as `ModelVisibleText` — supervisory messages
from typed events, and a **converse answer** as model-authored text passed through the
same redact→bound boundary (the model never holds a raw `send_message` tool). See
[`docs/telegram.md`](./docs/telegram.md) §8 and [`docs/chat.md`](./docs/chat.md).
**Tested by:** channel-routing + principal-trust tests; the converse-redaction red-team
(a secret in a model answer is redacted before the user sees it).

> **Amendment (v0.4.x, owner-authorized).** This law previously read "Telegram only by
> default." It is broadened to **any authorized, redacted channel** so a sanctioned
> conversational front door (`crustcore chat`; a future Telegram converse mode) can
> exist. The security property is unchanged and *strengthened* by being stated
> generally: one redacted, principal-authenticated boundary — the *channel* widened, the
> *boundary* did not.

### 16. The CLI is setup/admin/emergency; the only sanctioned conversational surface is `crustcore chat`

**Why:** avoids an *ungoverned* parallel control plane — the risk is *unaudited*
conversation, not conversation as such.
**Enforcement:** the base/nano CLI surfaces setup/admin/inspect/emergency commands only.
The conversational front door is the **explicit, feature-gated `crustcore chat`**
subcommand (non-nano `chat` feature; a stub in nano), which routes through the *same*
redact / principal-trust / policy / approval boundary as Telegram. It is a sanctioned,
typed channel — not a hidden one — and it cannot mint authority, approve a side effect,
or complete a task: the classifier grants nothing, persona/steering mint no authority,
and completion stays with the verifier and the human approval gate.
**Tested by:** CLI surface review; the chat pack's principal-trust, non-authoritative-
classifier, persona-cannot-authorize, and converse-redaction tests.

> **Amendment (v0.4.x, owner-authorized).** This law previously read "not a hidden
> second chat channel." A `crustcore chat` front door is now sanctioned — but it is the
> *opposite* of hidden: explicit, feature-gated, and routed through the same redacted,
> principal-trust, policy-gated boundary as every other surface. "No *ungoverned*
> parallel control plane" still holds.

### 17. Model/provider names are config and capability-probed, not permanent assumptions

**Why:** model availability changes; hard-coding rots and misroutes.
**Enforcement:** providers are probed; a dynamic local registry holds
capabilities. See [`docs/model-routing.md`](./docs/model-routing.md).
**Tested by:** router tests with a dynamic registry + provider-down fallback.

### 18. Self-improvement happens through PRs/evals, not live mutation of the running kernel

**Why:** a self-mutating kernel cannot be audited or trusted.
**Enforcement:** improvement is emitted as proposals/PRs gated by evals and a
contract-file gate; no live policy/sandbox/secret mutation. See
[`docs/self-improvement.md`](./docs/self-improvement.md).
**Tested by:** self-improvement gate tests (agent cannot weaken policy silently).

### 19. Nano feature build must remain below the configured size budget

**Why:** size is the flagship architectural claim.
**Enforcement:** CI size gate fails a PR that pushes `crustcore-nano` over budget
unless the maintainer explicitly raises the budget in the same PR.
**Tested by:** the nano size gate. See [`docs/nano-size-budget.md`](./docs/nano-size-budget.md).

### 20. Unused capabilities must cost zero model context and preferably zero linked code

**Why:** keeps the trusted core tiny and prompts lean.
**Enforcement:** capability packs are separate feature-gated crates/sidecars;
unused ones are not linked into nano and not loaded into context.
**Tested by:** `cargo tree`/`cargo bloat` checks + context-budget review.

## Red-team requirement

The following must each have a passing fixture before v0.1 (see
[`THREAT_MODEL.md`](./THREAT_MODEL.md) and [`ROADMAP.md` §19.3](./ROADMAP.md)):
repo file asks for token; issue comment says ignore policy; test output says
exfiltrate secret; MCP server returns hidden instructions; dependency
postinstall attempts network; external worker writes outside worktree; model
fabricates tool result; misleading approval text; sneaked GitHub workflow
modification; symlink escape; LD_PRELOAD/path-env escape.

A change that adds a new surface must add the corresponding red-team fixture in
the same PR.
